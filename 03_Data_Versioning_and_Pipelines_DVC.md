# Data Versioning and Pipelines (DVC)

**Tags:** #mlops #dvc #data-versioning #pipelines #experiments

## 🎯 Goal
Version control large datasets and machine learning models alongside code, build reproducible data processing pipelines (DAGs), and track ML experiments systematically without bloating the Git repository.

## 🛠 Technologies 
- **DVC (Data Version Control)**: Manages large files, datasets, and models by storing them in remote storage (S3, GCP, local network) while keeping lightweight pointer files (`.dvc`) in Git.
- **Git**: Works in tandem with DVC to version the code, DVC configuration files, and DVC pointers across different branches.

## 💻 Key commands 

### 1. Initialization and Tracking Basics
```bash 
# Initialize DVC in a Git repository
dvc init

# Track a large dataset or model (creates a .dvc file and updates .gitignore)
dvc add data/raw/dataset.csv

# Commit the DVC tracking files to Git
git add data/raw/.gitignore data/raw/dataset.csv.dvc
git commit -m "Track raw dataset with DVC"
```

### 2. Remote Storage Configuration
```bash
# Add a default remote storage (e.g., local folder, AWS S3, Google Drive)
dvc remote add -d myremote s3://my-dvc-bucket/project-data

# Push tracked data to the configured remote storage
dvc push

# Pull tracked data from the remote (used after cloning the repo or switching branches)
dvc pull
```

### 3. DVC Pipelines and Parameterization
```bash
# Create a pipeline stage (e.g., data processing) with dependencies, parameters, and outputs
dvc stage add -n process_data \
    -p process.chunk_size,process.threshold \
    -d src/process.py -d data/raw/dataset.csv \
    -o data/processed/features.csv \
    python src/process.py

# Execute the pipeline (DVC will automatically figure out what changed and what to run)
dvc repro
```

### 4. Pipeline Configuration (`dvc.yaml` & `params.yaml`)
```yaml
# dvc.yaml example
# This file is automatically generated by 'dvc stage add', but can be edited manually.
stages:
  process_data:
    cmd: python src/process.py
    deps:
      - src/process.py
      - data/raw/dataset.csv
    params:
      - process.chunk_size
      - process.threshold
    outs:
      - data/processed/features.csv

  train_model:
    cmd: python src/train.py
    deps:
      - src/train.py
      - data/processed/features.csv
    params:
      - train.epochs
      - train.learning_rate
    outs:
      - models/model.pt
    metrics:
      - metrics.json:
          cache: false
```
```yaml
# params.yaml example
# DVC automatically looks for this file to track parameters used in dvc.yaml
process:
  chunk_size: 1000
  threshold: 0.85

train:
  epochs: 50
  learning_rate: 0.01
```
```bash
# Run with
dvc repro
# It will automatically recognize `dvc.yaml` and `params.yaml`
# Run for more info
dvc repro --help 
```
### 5. Metrics and Experiments
```bash
# Run a new experiment without creating a permanent Git commit
dvc exp run

# Run an experiment while overriding a specific parameter from params.yaml
dvc exp run -S train.epochs=50 -S train.learning_rate=0.01

# Compare all tracked metrics across different experiments
dvc exp show
```
## 🧠 Conclusions and Problem Solutions

- **Data files are accidentally pushed to GitHub**: The actual `.csv` or `.pt` files were committed using `git add` instead of `dvc add`. **Solution**: Remove the files from Git cache using `git rm -r --cached data/`, ensure the DVC-generated `.gitignore` is committed, run `dvc add data/`, and then only commit the resulting `.dvc` files to Git.
- **`dvc pull` fails with an authentication or access error**: DVC cannot access the remote storage (like S3 or GCP). **Solution**: Ensure your cloud provider credentials are set in your environment variables, or use `dvc remote modify --local myremote credential_path <path>` to securely set authentication keys locally without committing them to `dvc/config`.
- **DVC pipeline runs entirely from scratch every time**: DVC does not recognize cached stages because dependencies (`-d`) or outputs (`-o`) were not properly defined in `dvc.yaml`. **Solution**: Review the `dvc.yaml` file. Ensure that every script, input file, and output file is strictly listed under `deps` and `outs` for each stage.
