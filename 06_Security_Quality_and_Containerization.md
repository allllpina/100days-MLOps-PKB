# Security, Data Quality, and Containerization 

Tags: #mlops #vault #great-expectations #docker #ci-cd
## 🎯 Goal
Secure sensitive credentials using a dedicated secrets manager, enforce strict data quality gates before training, and package ML environments into reproducible, lightweight containers for both training and serving.

## 🛠 Technologies 
- **HashiCorp Vault**: A centralized tool to securely store, manage, and control access to secrets (e.g., database passwords, API keys) rather than hardcoding them in `.env` files or Git.
- **Great Expectations (GX)**: A Python-based framework for data validation, profiling, and testing to ensure data pipelines output expected, high-quality data.
- **Docker**: A platform to build, ship, and run distributed applications in isolated containers. Multi-stage builds are used to minimize the final image size for serving.

## 💻 Key commands 

### 1. Secret Management (HashiCorp Vault)

Store a secret in Vault's Key/Value (KV) engine and authenticate programmatically using AppRole (designed for machines/services, not humans).
```bash 
# Write a secret (e.g., MLflow admin password) to a specific path
vault kv put secret/mlflow admin_password="SuperSecretPassword"

# Read a secret back manually
vault kv get secret/mlflow
```
```python
# Authenticate and fetch a secret in Python using hvac (Vault API client)
import hvac
import os

client = hvac.Client(url=os.environ['VAULT_ADDR'])
# Authenticate using AppRole credentials
client.auth.approle.login(
    role_id=os.environ['VAULT_ROLE_ID'],
    secret_id=os.environ['VAULT_SECRET_ID']
)

# Fetch the secret
secret = client.secrets.kv.v2.read_secret_version(path='mlflow')
db_password = secret['data']['data']['admin_password']
```

### 2. ### Data Validation (Great Expectations)

Define constraints on data (Expectations) and run them as Checkpoints. If a Checkpoint fails in a CI/CD pipeline, it acts as a blocking gate, preventing the model from training on corrupted data.
```bash
# Scaffold a new Great Expectations project structure
gx init

# Create a new Expectation Suite (a collection of rules)
gx expectation suite new user_data_suite

# Run a pre-configured Checkpoint to validate a dataset against the suite
gx checkpoint run validate_user_data
```
```python
# Example of authoring expectations via Python API
validator.expect_column_values_to_not_be_null(column="user_id")
validator.expect_column_values_to_be_between(column="age", min_value=18, max_value=99)
validator.save_expectation_suite()
```
### 3. ### Containerization (Docker for ML)

Build isolated environments. ML Serving requires multi-stage builds to strip out heavy build-time dependencies (like gcc or full python-dev packages), transferring only the compiled artifacts to the final, lightweight runtime image.
```dockerfile
# Example of a Multi-Stage Dockerfile for ML Serving
# Stage 1: Build dependencies (Heavy)
FROM python:3.12-slim AS builder
WORKDIR /app
COPY requirements.txt .
# Install build tools and compile wheels
RUN apt-get update && apt-get install -y gcc && \
    pip wheel --no-cache-dir --no-deps --wheel-dir /app/wheels -r requirements.txt

# Stage 2: Runtime environment (Lightweight)
FROM python:3.12-slim AS runtime
WORKDIR /app
# Copy only the compiled wheels from the builder stage
COPY --from=builder /app/wheels /wheels
COPY --from=builder /app/requirements.txt .
RUN pip install --no-cache /wheels/*
COPY . .
CMD ["mlflow", "models", "serve", "-m", "models:/fraud-detector@champion", "-p", "5000", "--host", "0.0.0.0"]
```
```bash
# Build the Docker image
docker build -t ml-serving:latest .
```

## 🧠 Conclusions and Problem Solutions

- **Vault: `PermissionDenied` when AppRole tries to read a secret**: The AppRole successfully authenticated, but its assigned policy does not grant `read` access to the specific KV path. **Solution**: Update the HashiCorp Vault policy (`.hcl` file) assigned to the AppRole to explicitly allow `capabilities = ["read"]` on the path `secret/data/mlflow`, then write the policy back to Vault.
- **GX: Checkpoint fails in the CI pipeline**: A data-quality test failed (e.g., finding unexpected NULL values or statistical drift in a new CSV), causing the script to exit with a non-zero status code. **Solution**: Review the Data Docs generated by GX. If the data is genuinely corrupted, the blocking gate worked as intended—fix the upstream data pipeline. If the data is correct but the rule was too strict, update the Expectation Suite thresholds.
- **Docker: Serving image is gigabytes in size**: Using standard `pip install` inside a single-stage `ubuntu` or `python` image includes unnecessary OS packages, caches, and build tools. **Solution**: Use a multi-stage Docker build (`AS builder` -> `AS runtime`) combined with `.dockerignore` to ensure the final image contains only the compiled dependencies and the minimal necessary runtime code.
